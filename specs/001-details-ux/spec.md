# Feature Specification: 今日の運航情報ページ UX改善

**Feature Branch**: `001-details-ux`
**Created**: 2026-03-08
**Status**: Draft
**Input**: User description: "今日の運航情報ページ（/details/today）の視認性・UX改善"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 運航ステータスを一目で確認する (Priority: P1)

フェリー利用者が今日の運航情報ページを開き、各便の運航状況（通常・遅延・欠航）を素早く把握したい。現状はプレーンテキストのみで色やバッジがないため、どれが欠航かを読まないとわからない。

**Why this priority**: 利用者が最も知りたいのは「今日の便は動いているか」の一点。ステータスの視覚化が最優先。

**Independent Test**: ステータスバッジ（色分け）のみ実装しても、欠航・遅延の便をページ上で即座に識別でき、ユーザーに価値を届けられる。

**Acceptance Scenarios**:

1. **Given** 欠航ステータスの運航情報が存在する, **When** ページを表示する, **Then** 該当便に赤いバッジ/ラベルで「欠航」と表示される
2. **Given** 遅延ステータスの運航情報が存在する, **When** ページを表示する, **Then** 該当便に黄色いバッジ/ラベルで「遅延」と表示される
3. **Given** 通常運航の情報が存在する, **When** ページを表示する, **Then** 該当便に緑のバッジ/ラベルで「通常運航」と表示される

---

### User Story 2 - 出発時刻順に便を確認する (Priority: P2)

フェリー利用者が今日乗る便を探す際、時系列で並んでいないと目的の便を探しにくい。出発時刻の早い便から順に表示されることで、次の便がすぐにわかる。

**Why this priority**: 情報の並び順はUXの基本。ステータス視認性の次に重要。

**Independent Test**: 複数の運航情報を持つ航路で、出発時刻昇順に並んで表示されることを確認できる。

**Acceptance Scenarios**:

1. **Given** 同一航路に複数の便がある, **When** ページを表示する, **Then** 便が出発時刻の早い順（昇順）に表示される
2. **Given** 便が1つしかない航路, **When** ページを表示する, **Then** 問題なくその1件が表示される

---

### User Story 3 - 今日の情報だけを確認する (Priority: P2)

フェリー利用者が「今日の運航情報」ページを開いたとき、明日以降の便が混在していると混乱する。今日の日付の便だけが表示されるべき。

**Why this priority**: ページタイトル「今日の運航情報」の約束を果たすための基本要件。

**Independent Test**: 今日と明日の運航データがある状態でページを開き、今日分のみ表示されることを確認できる。

**Acceptance Scenarios**:

1. **Given** 今日と明日の運航情報が両方存在する, **When** ページを表示する, **Then** 今日分の便のみが表示され、明日以降の便は表示されない
2. **Given** 今日の運航情報が存在しない, **When** ページを表示する, **Then** 「本日の運航情報はありません」というメッセージが表示される

---

### User Story 4 - スマートフォンで快適に確認する (Priority: P3)

フェリー乗り場でスマートフォンから運航状況を確認する利用者が、横スクロールなしに全情報を読める。

**Why this priority**: 実際の利用シーンではモバイル利用が多いが、機能面（P1/P2）が先決。

**Independent Test**: スマートフォン相当の画面幅（375px程度）でページを表示し、横スクロールなしに全情報が読めることを確認できる。

**Acceptance Scenarios**:

1. **Given** スマートフォン幅の画面, **When** ページを表示する, **Then** 横スクロールなしに全情報が1カラムで読める
2. **Given** 複数のフェリー会社・航路がある, **When** スマートフォンで表示する, **Then** 各カードが画面幅に収まって表示される

---

### Edge Cases

- 今日の運航情報がまったく存在しない場合、空リストではなくわかりやすいメッセージを表示する
- 備考（memo）フィールドが空・null の場合、備考欄ごと非表示にする（空のラベルが残らない）
- 出発時刻が null の場合、出発時刻の行ごと非表示にする。到着時刻が null の場合も同様に到着時刻の行ごと非表示にする
- フェリー会社は存在するが、当日の運航情報がない航路の場合、「本日の運航情報なし」と表示する

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: ページは今日の日付（operationDate = 今日）の運航情報のみを表示しなければならない
- **FR-002**: 各運航情報の運航ステータスを、通常（緑）・遅延（黄）・欠航（赤）の3色のバッジで視覚的に表示しなければならない
- **FR-003**: 同一航路内の運航情報は出発時刻（departureTime）の昇順で表示しなければならない
- **FR-004**: 備考（memo）が空またはnullの場合、備考欄は表示してはならない
- **FR-005**: ページはモバイル端末（画面幅375px以上）で横スクロールなしに表示できなければならない
- **FR-006**: 各運航情報はカード形式で表示し、出発時刻・到着時刻・ステータスを視覚的に目立つ位置に配置しなければならない
- **FR-007**: 「フェリー会社:」「航路:」「方向:」などの冗長なラベルプレフィックスを取り除き、情報をシンプルに表示しなければならない。航路の方向（direction）は航路名の直下にサブタイトルとして小さく表示しなければならない
- **FR-008**: 複数のフェリー会社が存在する場合、会社ごとのセクションを縦スクロールで並べて表示しなければならない

### Key Entities

- **Operation（運航情報）**: 特定日付・航路の1便分の運航情報。operationDate・departureTime・arrivalTime・status・memoを持つ
- **Route（航路）**: フェリーの航路。nameと方向（direction）を持ち、複数の運航情報（Operations）を持つ
- **Company（フェリー会社）**: 複数の航路を持つフェリー会社

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 利用者がページを開いてから3秒以内に、欠航・遅延便の存在を色で識別できる
- **SC-002**: 表示される運航情報がすべて今日の日付のものであり、明日以降のデータが含まれない（正確性100%）
- **SC-003**: スマートフォン（画面幅375px）での表示において横スクロールが発生しない
- **SC-004**: 備考が空の便において、空の備考ラベルが一切表示されない

## Clarifications

### Session 2026-03-08

- Q: 出発時刻・到着時刻が null の場合のフォールバック → A: 該当行ごと非表示にする

## Assumptions

- 運航ステータスは「通常（normal）」「遅延（delay）」「欠航（cancel）」の3種類が主要なケース
- 日本語ラベルはそのまま維持する（英語化などの国際化は対象外）
- CSSはプロジェクト既存の `styles.css` に追記する形で対応する
- サーバーサイドのタイムゾーンは日本時間（JST）で統一されている前提
- 認証・ログイン機能はこのフィーチャーの対象外
